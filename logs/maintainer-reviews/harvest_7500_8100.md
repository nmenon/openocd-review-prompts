> "Let me start saying that I agree with the purpose of this patch to have a `force` flag to flush
> and/or invalidate the register cache.
>
> Then, I have spent some time to check the use of `reg_arch_type::get() & set()` in the different
> targets. Quite a complicated situation.
>
> Every target has its own implementation, sometimes even two different implementations in the same
> target (see on cortex-m `armv7m_reg_type` and `dwt_reg_type`).
> On riscv both get&set read/write directly in the target; on cortex-m get() reads from target while
> set() only updates the register cache; on stm8 both get&set work on the register cache only!
> This probably explains the results from Marc's tests.
>
> I think that adding the new API `flush` and `invalidate`, without uniforming the targets, risks to
> make the implementation diverging and/or even more confused.
>
> Plus, I read in Jan's comment that 'The cache is here only for performance optimization'. Not true!
> On Cortex-A and AArch64 the register R0 is used to read/write either the other registers and the
> memory. And for wider data (e.g. floating point) both R0 and R1 are used. The register cache keeps
> the old values of R0 and R1 when they get dirty during the operation above.
> So the implementation of `target_flush_all_regs_default()` is broken for Cortex-A and AArch64
> because it flushes R0 as first, but then flushing the other registers it causes R0 to become dirty!
> At least these two targets need a dedicated implementation.
>
> I think we need a cleanup in the handling of registers and register's cache.
> I would like to see more centralized and common code in `target.c` and `register.c`, reducing the
> target specific code.
>
> Clearly, I cannot request you to fix all the targets. I don't think anyone here (me included) has
> the knowledge on all the supported targets, nor has the HW to test all the targets.
>
> But, could we approach this differently?
>
> What about starting with a first patch to clearly document the API we plan to introduce and their
> expected behavior?
> In `doc/manual/target.txt` we have an empty chapter `targetregister`.
> In `src/target/register.h` there is not a single doxygen comment.

## Patterns Observed

### 1. CMD_ARGC Checking — Exact Condition Required
Antonio Borneo consistently requires that COMMAND_HANDLER functions check CMD_ARGC with exact
conditions: `if (CMD_ARGC != N)` or `if (CMD_ARGC != 3 && CMD_ARGC != 4)` when the command
accepts a range of valid argument counts. He identifies the exact expected check pattern in
inline comments (seen multiple times: changes 7554, 8000).

### 2. Variable Declaration Style — Ongoing Disagreement
In change 8060, Marc Schink enforces "declare variables at first use" from the OpenOCD coding style
guide (https://openocd.org/doc/doxygen/html/stylec.html). Tomas Vanek disagrees for function-wide
variables (target, retval) but agrees for loop/temporary variables. Marc Schink's response:
consistency matters more than readability preference; the style guide exists to prevent per-patch
style debates that waste review time.

### 3. __attribute__((unused)) for Deliberately Unused Parameters
When a parameter becomes unused after a patch but should be kept for API/signature consistency,
Antonio Borneo requires `__attribute__ ((unused))` rather than removing the parameter (change 7569).
This pattern applies to functions like GDB packet handlers where all handlers have the same signature.

### 4. Commit Message Must Explain Why
Small CI/build fix commits must explain the exact failure and why the fix works. Antonio Borneo
rejected change 7551 PS1 because the commit message only said "Fixes Error Unable to find version
v0.2.0" without explaining what that error is, where it occurs, or why the version bump fixes it.
After the author added a failure log URL, Antonio approved.

### 5. Submodule Version Discipline
In change 7551, the patch incorrectly bumped a GitHub Actions action version while simultaneously
reverting the jimtcl submodule pointer. Antonio immediately spotted and rejected the submodule
regression as a -1, independent of the stated fix.

### 6. Integer Type Correctness
Marc Schink requires using the correct integer type matching the return type of called functions.
Specifically: `timeval_ms()` returns `int64_t`, so variables capturing its return value must be
`int64_t`, not `int`, `long`, or `uint32_t` (change 8060, line 255).

### 7. Unsigned vs Signed for Counts/Sizes
Marc Schink flags signed integer types for values that represent counts, sizes, or indices that
cannot be negative. CMD_ARGC-derived values and loop counters should be `unsigned` or explicit
size types (change 8060, line 1079).

### 8. CMD_ARGC Check Before Using Arguments
Marc Schink (change 8060, line 1079) requires that variable declarations/initializations that
depend on argument count come AFTER the CMD_ARGC check. Declaring `unsigned bank = CMD_ARGV[0]`
before `if (CMD_ARGC < 1) return ERROR_COMMAND_SYNTAX_ERROR` is wrong.

### 9. Useless/Dead Code Detection
Marc Schink explicitly calls out statements that are logically useless: assignments whose value
is immediately overwritten, or conditions that are always true/false in context (change 8060,
lines 620, 1119). These are not just style issues but logic errors or code rot.

### 10. Documentation-First for API Changes
For any change that adds or modifies public APIs (especially in target register handling), Antonio
Borneo requests a documentation patch first to define the API contract before implementation.
He specifically cites the empty `doc/manual/target.txt#targetregister` chapter and the absence
of doxygen in `src/target/register.h` as evidence that the API is not well-defined (change 8070).

### 11. Cross-Architecture Correctness
Antonio Borneo deeply analyzes cross-target behavior. For register flushing (change 8070), he
discovered that `target_flush_all_regs_default()` is architecturally broken on Cortex-A/AArch64
because R0 is used as a scratch register during flush — flushing R0 first then dirtying it again
when flushing other registers creates a race. He expects implementors to understand these
architectural constraints, not just apply a fix to one target.

### 12. Register Cache Semantics
Antonio Borneo clarifies that the register cache is NOT merely a performance optimization
(contra what the patch author assumed). For Cortex-A/AArch64, the cache preserves R0/R1 values
corrupted by register read/write operations that use R0/R1 as intermediaries. Any flush/invalidate
API must account for this semantic distinction.

### 13. Reviewer Collaboration — Peer Check Requests
Antonio Borneo explicitly asks other maintainers (e.g., Tomas Vanek) to review his own patches
to avoid self-review bias, particularly for obscure code he cannot test (change 7599: "I was just
asking for a second check, to avoid any bias on my side. I cannot test this old code").

### 14. Behavior/Documentation Mismatch Flagging
Tomas Vanek flags pre-existing behavior-documentation mismatches when COMMAND_HANDLER rewrites
happen (change 7554). Even though the rewrite is not supposed to change behavior, the rewrite
is a good opportunity to fix such mismatches. If not fixed, they should be explicitly noted.

### 15. Review Communication Tone
The Tomas Vanek / Antonio Borneo exchange on change 7575 shows that frustrated tone occasionally
appears: "Antonio, please stop inventing your 'why not's or this series never gets finished." This
occurs in the context of a large patch series where the author is tired of scope-creep objections.
Marc Schink acknowledges limited time: "sorry for the late response - my time is limited and as
you probably know I only do all this on the side and not full-time."
