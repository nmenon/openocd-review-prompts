### Change 8130 — src/helper/options.c: Extend IS_DARWIN guard (NEW/open)
Author: Sevan Janiyan

**Antonio Borneo:**
> "I think it could be simpler and more readable if you add a line in the file configure.ac: `AC_CHECK_HEADERS([libproc.h])` between the other AC_CHECK_HEADERS directives. This will add a new macro HAVE_LIBPROC_H in the generated config.h. Then in the file you can use `#ifdef HAVE_LIBPROC_H ... #endif`. Does this work properly on MacOS?"

> "Of course it should be: `#if IS_DARWIN #ifdef HAVE_LIBPROC_H #include <libproc.h> #endif #endif` ... Repeating the MacOS version check over and over makes the code unreadable."

**Rule derived:** Platform-specific headers should use AC_CHECK_HEADERS() in configure.ac, generating HAVE_<HEADER>_H macros, rather than repeated inline version/platform guards (HP-CFG-1).

## Patterns by Reviewer

### Antonio Borneo — Recurring Themes
1. **Commit message format**: Subject must be single line + blank line + body. Flags multi-line subjects.
2. **TCL dead code**: Aggressively removes: unused variables, dead procs, unreachable code, already-default settings, stale comments.
3. **configure.ac for portability**: Use AC_CHECK_HEADERS/AC_CHECK_FUNCS → HAVE_* macros instead of inline platform version guards.
4. **Late independent override**: Will give -1 after another maintainer's +2 if he finds real issues (8232). Review is always independent.
5. **Constructive blocking**: When blocking (-1), provides specific alternative implementation, not just "fix it". (8130: provided exact autoconf code; 8127: referenced GTK debug model).
6. **Questions non-default settings**: "Why do you need to enable them? What issue do you get?" for non-default GDB/target settings.
7. **Verbosity consensus**: Against adding state to every log line without per-area verbosity control.

### Tomas Vanek — Recurring Themes
1. **Long-term patch stewardship**: Picks up stalled patches and drives them to completion (8202: 1.5yr WIP).
2. **Practical interim fixes**: Accepts zero-initialization + FIXME comment when full fix is too large for one patch (8181).
3. **Feature proposals in reviews**: Suggests architectural improvements rather than just accepting patches as-is (8127: named debug areas proposal).
4. **Verbosity gatekeeper**: Against unnecessary log noise; wants per-area verbosity control.
5. **Grants +2 with acknowledgement comments**: Style of closing out a patch with brief note on what was fixed.

### Marc Schink — Recurring Themes
1. **User experience focus**: Primary question is "how does this help regular users?" before accepting.
2. **Misleading output**: Will block on output that confuses users even if technically correct (8127: "<unknown>" state label).
3. **Asks for use case justification**: "Can you explain in what situations this is useful for regular users?"
4. **High standards applied to own work**: His own patches (8256) sail through, implying he self-enforces what he demands.

### Cross-reviewer Consensus (all 3 agree)
- **Log verbosity**: Strongly against adding state to every log line. All three independently objected (8127). Independent verbosity control needed.
- **Dead code**: All enforce clean .cfg files; Borneo most active, but others agree.
- **Commit message format**: All enforce the single-line-subject + blank-line + body format.
